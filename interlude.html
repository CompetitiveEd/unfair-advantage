The "Placemat" Visualizer

SLIDE 4 PLAYBACK

Executives don't read 50 pages. They look at one diagram. This prompt generates the "One Page" that validates their pain and sells your future state instantly.

E-01: VISUAL SPECIFICATION GENERATOR
ROLE: You are a Strategic Visualiser and Enterprise Architect. Your goal is to synthesise a complex transformation into a SINGLE visual "placemat" that an executive can understand in 30 seconds.

INPUTS:
- Client Current State (from RFP/Research): [PASTE]
- Client Goals/Future State: [PASTE]
- Key timelines and constraints: [PASTE]

Produce a VISUAL SPECIFICATION for a "Playback" slide:

1. THE "FROM" STATE (Visual Context)
   Describe the visual metaphor for their current pain.
   (e.g., "Fragmented silos," "Tangled wires," "Disconnected regions").
   List the specific data points to label (e.g., "28 Domains," "Inconsistent UX").

2. THE "TO" STATE (Visual Goal)
   Describe the visual metaphor for the solution.
   (e.g., "Unified hub," "Streamlined highway," "Single Identity").
   List the specific outcomes to label (e.g., "Single Domain," "Reduced Opex").

3. THE BRIDGE (The Transformation)
   How do we visually represent the journey?
   (e.g., "Phased migration stream," "The Discovery Filter").
   What are the 3-4 distinct phases to show on the arrow/timeline?

4. THE "YES" LABELS
   What specific client terms must appear on the diagram to make them nod?
   (e.g., if they call it "Project Voyager," the diagram must use that tag).

5. CAPTIONING
   Write the single sentence caption that goes at the bottom.
   Must summarize the ROI (e.g., "Consolidating 28 domains to deliver single-identity security and 30% opex reduction by 2018").

WHY THIS MATTERS â€” Most proposals fail because they describe the process, not the transformation. This prompt forces you to define the From and To states clearly. Use the output to brief your graphic designer or build a clean PowerPoint slide.

The Logic Engine

SLIDE 23 COMPLEXITY MATRIX

Most bids fail because they treat all servers, users, or apps the same. This prompt forces the AI to categorize complexity (Low/Med/High), ensuring you price for reality, not hope.

E-02: LOGIC & PRICING MODEL
ROLE: You are a Technical Solution Architect and Risk Analyst. You are designing the logic for a massive migration/transformation project. Generic pricing will kill this bid. We need a "Decision Logic Matrix" that categorizes complexity.

INPUTS:
- RFP Technical Scope / Requirements: [PASTE]
- Asset data (if available): [PASTE]
- Known constraints: [PASTE]

Produce a LOGIC ENGINE:

1. COMPLEXITY DRIVERS
   Identify the 5-7 technical factors that make a unit (server/app/user) hard to migrate.
   (e.g., Hard-coded IPs, Legacy OS, Third-party dependencies, Physical dongles).

2. THE DECISION TREE
   Create a logic flow:
   - IF [Condition A] THEN [Action B] (Cost = X)
   - IF [Condition C] THEN [Action D] (Cost = 3X)
   Example: "Is source code available? If NO -> Re-platforming required -> High Cost."

3. CLASSIFICATION MATRIX (The "T-Shirt Sizing")
   Define criteria for:
   - SIMPLE (Standard process, automated)
   - MEDIUM (Requires minor remediation)
   - COMPLEX (Requires manual re-architecture)
   - TOXIC (Out of scope / separate T&M project)

4. SCORING MECHANISM
   Assign a "risk score" to the drivers.
   (e.g., Base = 1. Hard-coded IP = +5. No Documentation = +10).
   This allows us to model the "unknowns" mathematically.

The Commercial Shield

SLIDE 46 "SILENT KILLERS"

Winning the bid is useless if you lose money on the delivery. This prompt generates the commercial armour that protects your margin from "scope creep" and "data migration black holes."

E-03: COMMERCIAL RISK SHIELD
ROLE: You are a Commercial Risk Officer and Senior Bid Lead. Your job is to protect the firm from "Black Hole" risksâ€”requirements that look simple but have infinite cost depth (like "migrate all data" or "fix all legacy apps").

INPUTS:
- RFP Scope of Work: [PASTE]
- Technical Constraints (from Logic Engine): [PASTE]

Produce the COMMERCIAL SHIELD:

1. THE "SILENT KILLERS" LIST
   Identify 3-5 requirements in the RFP that look standard but carry massive hidden risk.
   (e.g., "Mailbox size 2GB" -> implies massive SAN upgrade if not tiered. "All applications" -> implies owning code remediation for 20-year-old apps).

2. DEFENSIVE DEFINITIONS (The "In/Out" Table)
   Draft the specific text for:
   - IN SCOPE (What we strictly control)
   - OUT OF SCOPE (The dangerous stuff: App upgrades, licensing, physical moves)
   - DEPENDENCIES (What the client MUST do for the price to hold)

3. THE "CONSULTATIVE STATUS" DISCLAIMER
   Draft a clause that defines our pricing as "Indicative based on assumptions" rather than "Fixed Price."
   Crucial phrase: "Only a detailed discovery phase will allow accurate cost detail."

4. ASSUMPTION LOG
   List the 5 critical assumptions that, if false, trigger a Change Control.
   (e.g., "Assumes administrative access is provided within 48 hours").
!

Warning: Never submit a fixed price without the "Commercial Shield" outputs. Use the "Out of Scope" list to define the boundary of your liability.
